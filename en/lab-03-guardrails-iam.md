# Lab 03: AI Governance Scanner — Automatic IAM Compliance Audit for Bedrock

> 🇪🇸 [Versión en español](../lab-03-guardrails-iam.md)

**Team:** 2–3 people
**Level:** Intermediate–Advanced

---

## 1. Lab name

**AI Governance Scanner** — Automatic IAM compliance audit for Bedrock

---

## 2. Main skill

**AI Governance and cloud infrastructure security** — IAM policy analysis, compliance gap detection, and generation of actionable remediations to enforce AI governance.

---

## 3. Problem to solve

AI governance policies implemented "by code convention" don't scale. When a new team creates a microservice with Bedrock access and forgets to include the guardrail required by corporate policy, no one detects it until a manual audit or an incident.

The structural problem: IAM allows `bedrock:InvokeModel` without any condition key. If the corporate policy requires every call to include the `bedrock:GuardrailIdentifier` guardrail, that condition must be in the role's IAM policy — not in the microservice code. Without IAM-level enforcement, any deploy that forgets is an invisible compliance gap.

The team will build `ai-governance-scan`: a tool that makes compliance state visible in seconds and delivers ready-to-apply fixes.

---

## 4. Target product

An `ai-governance-scan` CLI that:

1. Enumerates all IAM roles in the account
2. Identifies which roles have `bedrock:InvokeModel` or `bedrock:Converse` permissions
3. Verifies whether they have the `bedrock:GuardrailIdentifier` condition with the required ARN
4. Classifies each role as `COMPLIANT` or `NON_COMPLIANT`
5. Generates the exact text of the corrective policy for each non-compliant role (copy-paste ready)

**Output:** HTML + JSON report with executive summary, per-role status, and ready-to-apply fixes.

**Suggested validations:**

- Run the scan over the test environment and verify it correctly classifies compliant and non-compliant roles
- Apply the corrective policy generated for a non-compliant role and re-run the scan to verify it now appears as compliant
- Test with roles that use wildcards (`bedrock:*`, `bedrock:Invoke*`) and verify the scanner detects them correctly
- Check that the HTML report includes fixes ready to copy and paste in the right format

*In the show & tell, the team shows the validations they ran and centers the conversation on decisions made, problems encountered, and lessons learned.*

---

## 5. MVP scope

**First phase — Scanner core:**
- Test environment setup: script that creates 10-15 IAM roles with different compliance levels
- Implement the scanner: enumerate roles, filter by Bedrock access, verify condition keys
- Output to JSON with the state of each role
- Generate corrective policy text for non-compliant roles

> **Reflection checkpoint:** Note which AI tools you used in this phase, on which tasks, and a moment where AI didn't help or caused problems.

**Second phase — HTML report and validation:**
- Jinja2 template for the HTML report with executive summary + role table + fixes section
- `--dry-run` command that shows the fixes without applying them
- Support for `--region` to scan a specific region
- Validation test: apply a fix generated by the scanner and confirm the re-scan detects it as compliant

> **Reflection checkpoint:** Note whether AI changed how you approached this phase vs. the first, and what you would adjust about AI use if starting again.

---

## 6. Extension

- Scan **multiple accounts** via AWS Organizations: iterate over all accounts in the org and produce a consolidated report
- `--auto-fix` mode: apply corrective policies directly with interactive per-role confirmation
- Extend the scanner to verify **SCPs (Service Control Policies)** at the OU level
- **Programmatic scan Lambda** that runs daily and publishes the result to S3, with notification if the number of non-compliant roles increases

**Non-AWS alternative (for teams without full IAM access):**
Build the same scanner targeting GitHub repos: search for LLM API calls (`bedrock_client.invoke_model`, `anthropic.messages.create`, `openai.ChatCompletion.create`) that aren't accompanied by required corporate headers or that have hardcoded API keys. Same structure: scanner + classification + suggested fixes.

## 7. Suggested stack

**AWS (recommended):**

| Component | Service / Library |
|---|---|
| Scanner | boto3: `iam.list_roles`, `iam.get_role_policy`, `iam.list_attached_role_policies`, `iam.get_policy_version` |
| CLI | Python 3.12 + Typer |
| Templating | Jinja2 |
| Output | HTML + JSON |

Estimated cost: **$0** — all operations are IAM API calls with no charge.

**Open-source alternative (repo scanner):**

| Component | Tool |
|---|---|
| Scanner | Python with `ast` (Python) or regex (Node/Java) |
| Target | Local repos or GitHub API |
| Output | Same HTML + JSON format |

## 8. Terrain to explore

- How do you detect access to `bedrock:InvokeModel` when the policy uses wildcards like `bedrock:*` or `bedrock:Invoke*`? What other equivalent patterns exist?
- What does "compliant" mean in IAM terms for Bedrock? When is a broad permission acceptable vs. problematic?
- How do you generate a remediation policy that's minimally permissive without breaking the existing workload?
- What happens when `list_roles` returns thousands of roles without pagination? How is information silently lost?
- What are the practical differences between inline and managed attached policies when auditing them programmatically?
- How do you validate that an AI-generated policy correction is safe before applying it in a real account?
- What makes a test environment representative enough to trust the scanner's results?
- Was AI useful for writing the IAM policy parser with its edge cases (wildcards, NotAction, multiple statements), or did it produce code that only covered the happy path?
- How did the team use AI to generate corrective policies? Were there cases where AI generated syntactically valid but semantically incorrect fixes?
